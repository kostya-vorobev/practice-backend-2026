# Расширенная схема работы по 5 этапам

Этот документ описывает, как проходить каждый этап практики: что делать, в какой последовательности, что должно получиться в итоге и как себя проверить.

Схема общая для **Booking API** и **Survey API**. Отличается только предметная логика.

---

## Этап 1. Проектирование и старт

### Цель

Собрать основу проекта: понятная модель данных, черновой контракт API и рабочая структура репозитория.

### Последовательность работы

1. Зафиксируйте границы задачи.
   - Что является основными сущностями проекта.
   - Какие действия доступны ролям пользователей.
   - Какие ограничения критичны для предметной области.
2. Спроектируйте модель данных.
   - Определите таблицы и связи между ними.
   - Для каждой сущности определите обязательные поля, статусы и ограничения.
   - Отдельно проверьте участки с повышенным риском:
     - Booking API: пересечения интервалов времени и статус бронирования.
     - Survey API: типы вопросов, варианты ответов, структура пользовательских ответов.
   - Описать допустимые переходы состояний, запреты и ошибки для недопустимых переходов. (например Survey: draft -> published -> closed, запрещено редактировать структуру после published)
3. Опишите API-контракт до начала кодирования.
   - Список эндпоинтов, назначение, роли доступа.
   - Форматы входных и выходных данных.
   - Ошибки валидации и отказа доступа.
4. Подготовьте каркас репозитория и проекта.
   - Базовая структура каталогов.
   - Базовые конфигурации окружения.
   - Заготовка README с описанием выбранной предметной области.
5. Сверьте модель данных и API между собой.
   - Любой эндпоинт должен опираться на реальные сущности и связи.
   - Любая критичная сущность должна быть отражена в API.

### Выход этапа

- ER-диаграмма.
- Роли и права (матрица - Действие/Роль1/Роль2/Роль3) (не обязательно, помогает при поэтапной работе, хорош как самостоятельный артефакт, на самом деле спрятан в уточненом виде в пункте "Черновой список эндпоинтов")
- Черновой список эндпоинтов (Endpoint=метод+путь, Кто Может=Роли, Условия=напрмер Админ видит все, Пользователь - только своё).
- Готовая структура репозитория.
- Миграции/схема БД, согласованные с ER-диаграммой.
- README с описанием проекта. (предыдущие описательные пункты можно вложить сюда, но можно и отдельными файлами)

### Критерии

- Логика ролей и доступа понятна без чтения кода.
- Для каждой основной сущности есть полный жизненный цикл.
- Не осталось "неопределённых" полей и связей в модели.

## Этап 2. Авторизация и базовый CRUD

### Цель

Запустить "скелет" продукта: доступ по ролям, базовые операции с ключевыми сущностями и строгая валидация.

### Последовательность работы

1. Реализуйте контур аутентификации и авторизации.
   - Вход/регистрация.
   - Получение и проверка токена.
   - Разделение доступа по ролям.
2. Реализуйте CRUD для базовых сущностей этапа.
   - Booking API: ресурсы и их характеристики.
   - Survey API: опросы, вопросы, варианты ответов.
3. Сразу внедряйте валидацию на входе.
   - Типы и диапазоны значений.
   - Обязательные поля.
   - Бизнес-ограничения предметной области.
4. Продумайте ошибки и ответы API.
   - Единый формат ошибок.
   - Ясные причины отказа (доступ, валидация, отсутствие данных).
5. Подготовьте тестовые данные для демонстраций.
   - Наборы пользователей/ролей.
   - Наборы сущностей для быстрых сценариев проверки.

### Выход этапа

- Рабочая авторизация с разграничением доступа.
- Базовый CRUD по ключевым сущностям.
- Валидация входных данных.
- Набор тестовых данных для демонстрации (Seeder-ы).

### Критерии

- Операции не выполняются без авторизации там, где она обязательна.
- Ролевая модель действительно ограничивает доступ.
- Ошибки валидации объясняют, что именно не так.
- Демонстрация при помощи подготовленной Постман коллекции
- Необязательно:
  - Тесты на негативные сценарии (удобно для фиксации поведения)

## Этап 3. Основная бизнес-логика

### Цель

Реализовать ключевую функционал продукта, защитить систему от критичных некорректных сценариев.

### Последовательность работы

1. Выделите главные бизнес-правила и формализуйте их.
   - Booking API: конфликт интервалов, правила отмены, доступ к чужим бронированиям.
   - Survey API: жизненный цикл статусов, валидация ответов по типу вопроса, запрет повторного прохождения.
2. Реализуйте happy path.
   - Сначала обеспечьте корректный основной сценарий.
   - Затем добавляйте защиту от нарушений.
3. Закройте негативные сценарии.
   - Конфликтные данные.
   - Недопустимые переходы статусов.
   - Повторные и запрещённые действия.
4. Добавьте журналирование на уровне бизнес-операций.
   - Что произошло.
   - Кто инициировал действие.
   - Почему действие было отклонено (если отклонено).
5. Проведите сценарную проверку end-to-end.
   - Полный пользовательский поток от входа до ключевого результата.

### Выход этапа

- Ключевая бизнес-логика работает и защищена от типовых нарушений.
- Негативные сценарии корректно отклоняются.
- Пользователь может пройти основной процесс от начала до конца.

### Критерии

- Нельзя "обойти" ограничения предметной области.
- Основные ограничения реализованы явно, а не "случайно".
- Поведение системы предсказуемо для одинаковых входных условий.
- Демонстрация при помощи подготовленной Постман коллекции

## Этап 4. Продвинутый функционал

### Цель

Добавить функции, которые делают API практичным в реальном использовании: поиск, аналитика, фильтрация, удобство работы со списками.

### Последовательность работы

1. Сделайте обязательные функции этапа 4 для своего проекта.
   - Booking API: расписание, поиск свободных ресурсов, фильтры, отзывы и средний рейтинг.
   - Survey API: аналитика, экспорт результатов, фильтры и сортировка опросов.
2. Реализуйте сначала базовый рабочий вариант.
   - Сначала добейтесь правильного результата на небольших данных.
   - Оптимизацию оставьте на потом.
3. Добавьте единые правила для всех списков.
   - Пагинация.
   - Фильтрация.
   - Сортировка.
4. Проверьте несколько понятных сценариев вручную.
   - Успешный сценарий.
   - Сценарий с фильтрацией/поиском.
   - Сценарий с пустым результатом.
5. Подготовьте короткое демо через Postman-коллекцию.
   - Что показываете по шагам.
   - Какие данные для этого заранее подготовлены.

### Выход этапа

- Реализованы все обязательные функции
- Пагинация, фильтрация и сортировка работают по единым правилам.
- Подготовлен понятный демо-сценарий и коллекция для показа.

### Критерии

- Все обязательные сценарии этапа 4 работают стабильно.
- Ответы API предсказуемы и понятны для проверки.
- Демонстрация проходит по подготовленной Postman-коллекции.

## Этап 5. Тесты, документация, упаковка

### Цель

Подтвердить надёжность решения и сделать проект воспроизводимым для проверки.

### Последовательность работы

1. Определите минимальный набор критичных тестов.
   - Бизнес-правила высокой важности.
   - Доступ по ролям.
   - Негативные сценарии, которые часто ломают систему.
2. Оформите API-документацию.
   - Что делает каждый эндпоинт.
   - Какие данные принимает и возвращает.
   - Какие ошибки возможны и при каких условиях.
3. Подготовьте контейнеризацию и воспроизводимое окружение.
   - Приложение и база данных.
   - Порядок инициализации.
   - Принцип запуска проекта "из коробки".
4. Доведите README до уровня "проверяющий понял всё за 5 минут".
   - Цель проекта.
   - Реализованные возможности.
   - Как проверить ключевые сценарии.
5. Проведите финальную самопроверку по чеклисту курса.
   - Сверьте требования к сдаче и фактическую реализацию.
   - Убедитесь, что демонстрация проходит без ручных "доправок".

### Выход этапа

- Автотесты на критичные сценарии.
- Полная документация API.
- Воспроизводимый запуск в контейнерном окружении.
- Финальный README для проверки и защиты.

### Критерии

- Проект можно проверить по документации без дополнительных пояснений.
- Критичные бизнес-правила покрыты тестами.
- Финальная демонстрация повторяема и стабильна.

---

## Рекомендуемый ритм работы внутри каждого этапа

1. Разбить этап на небольшие задачи.
2. Реализовать основной сценарий.
3. Закрыть ошибки и негативные ветки.
4. Зафиксировать результат в документации и подготовить демонстрацию.
